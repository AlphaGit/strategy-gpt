## Context

The Diátaxis "Tutorials" quadrant on the docs site holds three pages today (`first-backtest.md`, `authoring-a-strategy.md`, `running-an-optimization.md`). The hypothesize loop already has reference (`docs/reference/hypothesize-cli.md`) and how-to (`docs/how-to/run-hypothesize.md`) coverage, but a learn-by-doing walkthrough is missing.

Two CLI realities constrain the design:

1. `strategy-gpt hypothesize <strategy>` currently halts at `wiring_incomplete` unless `--dry-run` is set — full end-to-end execution requires operator-specific KB + engine collaborators wired in Python. The tutorial therefore cannot demonstrate a real loop run via the CLI alone.
2. `strategy-gpt hypothesis replay <decision_id>` and `strategy-gpt hypothesis diff <decision_id>` read from a per-strategy ledger that must already contain a recorded decision. Without that ledger present in a fresh clone, the walkthrough has nothing to replay against.

The tutorial spec (`openspec/specs/documentation/spec.md`) requires bundled-reference or shipped-stub data — no external services without a documented fallback.

## Goals / Non-Goals

**Goals:**

- Land one new page, `docs/tutorials/hypothesize-loop.md`, exercising `hypothesize --dry-run`, `hypothesis replay`, and `hypothesis diff` end-to-end against fixture data already in the repo (or generated by a one-shot Python command shipped with the repo).
- Conform to the five-section tutorial skeleton enforced by `documentation` spec.
- Add the page to `docs/tutorials/index.md` and `mkdocs.yml`.
- Promote the `documentation` spec's "initial tutorials" requirement from three to four pages.

**Non-Goals:**

- Re-architect the `hypothesize` CLI to run end-to-end without Python wiring. The `wiring_incomplete` state is documented; the tutorial works within it.
- Document the Python `HypothesizeDeps` entrypoint — that lives in the how-to (`docs/how-to/run-hypothesize.md`) and reference quadrants and should not migrate into Tutorials.
- Ship a frozen recorded fixture for the hypothesis ledger. The simpler choice is to drive `python -m strategy_gpt.smoke --write` (or equivalent) at tutorial step 1 so the per-strategy ledger and decision rows are generated from the in-tree stubs.

## Decisions

**1. Fixture source: generate, do not freeze.**
The walkthrough's `replay`/`diff` steps need at least one recorded decision under `ledger/strategies/<strategy>/`. Two options:
- *(a)* Ship a pre-recorded ledger directory under `kb/fixtures/hypothesis_ledger/` and tell the reader to copy it.
- *(b)* Run `python -m strategy_gpt.smoke --write <path>` at step 1 of the walkthrough; the smoke driver already exists (`python/strategy_gpt/smoke.py`), uses pure stubs (no LLM keys), and produces a representative `HypothesizeResult`.

Pick *(b)*. Pros: zero new fixture files to maintain; the generator path is already exercised by `test_smoke.py`, so drift is detected in CI. Cons: requires a small CLI surface from `strategy_gpt.smoke` to emit ledger rows (it currently emits a JSON report). If that surface doesn't yet exist, the tutorial implementation must extend the smoke driver to optionally write to a `--ledger-root` so `replay` and `diff` have something to read. The extension is small (call `PerStrategyLedger.append_*` after `hypothesize()` returns) and lives in `strategy_gpt/smoke.py`, not in CLI hot paths.

**2. CLI step 1 is `--dry-run`, not a full run.**
Showing the reader the `wiring_incomplete` JSON branch would be confusing. `--dry-run` echoes a clean resolved-flags summary, satisfies the "command + expected output" tutorial rule, and the page can call out (in `What next`) that full runs go via Python per the how-to.

**3. Documentation spec amendment: MODIFIED, not ADDED.**
The existing "ships at least three initial tutorials" requirement already governs the named-set. Promoting the count to four (and naming the fourth) is a *modification* of that requirement, not a new one. Spec delta uses `## MODIFIED Requirements` with the full updated requirement block per OpenSpec rules.

**4. Walkthrough order:**
1. Generate fixture ledger via `python -m strategy_gpt.smoke --write-ledger ledger`.
2. `strategy-gpt hypothesize vxx_volatility_range --dry-run` — input-validation smoke test, prints resolved flags JSON.
3. List recorded decisions (`ls ledger/strategies/vxx_volatility_range/`) so the reader has a real `<decision_id>` to use.
4. `strategy-gpt hypothesis replay <decision_id>` — reconstruct files-set bundle, print summary JSON.
5. `strategy-gpt hypothesis diff <decision_id>` — print unified diff vs baseline source bundle.

**5. Tutorial uses the bundled VXX strategy name.**
The `documentation` spec requires bundled reference or stub data. The smoke driver records under a stub strategy name (`smoke_strategy` or similar in `smoke.py`). If the recorded strategy name diverges from `vxx_volatility_range`, the tutorial uses whatever name the smoke driver writes — the page documents the actual fixture, not an aspirational one. This is decided at implementation time after inspecting `smoke.py`.

## Risks / Trade-offs

- *Risk:* The smoke driver does not currently persist a per-strategy ledger; only emits a JSON report. *Mitigation:* extend `strategy_gpt.smoke` with a `--ledger-root` option that drives `PerStrategyLedger.append_hypothesis` and `append_decision` after the in-memory run. Scope is contained to the smoke module.
- *Risk:* Output drift — a future change to the smoke driver alters the JSON shape and breaks the expected-output snippets in the tutorial. *Mitigation:* the existing `test_smoke.py` regression fixture already pins shape; the tutorial's expected-output blocks can use the same `// ...` truncation pattern as `running-an-optimization.md` so cosmetic drift is tolerated.
- *Risk:* `mkdocs build --strict` fails on a broken cross-link in `What next`. *Mitigation:* link only to pages that already exist (`docs/how-to/run-hypothesize.md`, `docs/reference/hypothesize-cli.md`, etc.).
- *Trade-off:* The tutorial demonstrates the *CLI surface* of the hypothesize loop, not the *outcome* of a loop run. Operators wanting a real loop result are punted to the how-to. The tutorial spec accepts this so long as commands run and outputs match.
